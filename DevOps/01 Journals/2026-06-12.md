## Quá trình (Claude làm Reasoning Engine)

**Vấn đề "works on my machine":**
- App chạy ổn trên dev, lỗi trên production vì kiến trúc máy / OS / môi trường khác nhau.
- Giải pháp: Docker.

**Docker image là gì:**
- Ban đầu hiểu image là "danh sách yêu cầu về môi trường" (sai).
- Qua Socratic probing (tắt internet vẫn chạy được → Node 18 lấy từ đâu?), rút ra: image chứa luôn các file thực tế — OS libraries, runtime binary, app code — không phải danh sách.
- Image = filesystem snapshot đóng gói sẵn. Container = instance đang chạy từ image đó.

**Kubernetes cơ bản:**
- Hiểu sai: nghĩ Pod quản lý các máy.
- Sau khi giải thích trực tiếp (do bế tắc): Container → Pod → Node → Cluster.
- Pod: bọc 1+ container, chúng chung một IP.
- Node: máy vật lý/VM chạy nhiều Pod.
- Cluster: tập hợp nhiều Node.
- Kubernetes: orchestrator — khi Node chết, tự dời Pod sang Node còn sống.

## Explain-back (Feynman Check)
- **Docker image**: đạt mechanism-level sau probing (offline run test). `Partial` → gần `Understood`.
- **Kubernetes Pod rescheduling**: "nó lấy pod từ máy chết sang node khác để chạy" — đúng cơ chế. `Partial`.
- **Kubernetes hierarchy**: nhận sau giải thích trực tiếp, chưa tự suy luận được. `Partial`.

## Misconceptions & Gaps (Ghi nhận lỗi)
- **[GAP]** Tưởng Pod là thứ quản lý Node/máy — thực ra ngược lại: Node là máy chứa Pod.
- **[GAP]** Tưởng Docker image là danh sách yêu cầu (như package.json) — thực ra là filesystem snapshot đầy đủ.
