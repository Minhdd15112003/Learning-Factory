## Khái niệm hôm nay
### [[Polymorphism]] — Dynamic dispatch — `Understood` (Good)
Dạy challenge-first từ chính code hôm qua (`Context` giữ `INotify` nhưng chạy `send()` của Email).

- **Điểm mấu chốt bạn tự sửa:** ban đầu nói "gắn với Email từ lúc compile". Qua ví dụ `userInput` (compiler không thể biết trước Email hay Sms), bạn tự đảo sang **runtime** — đúng bản chất dynamic dispatch.
- **Nửa còn lại (vai trò compiler):** bạn nêu đúng — compiler soi **static type** `INotify` để gác method nào *gọi được*; `sendBulk()` không có trong interface nên bị chặn dù vật thể Email có thật.
- **Cặp thuật ngữ chốt:** static type (compiler, compile-time) = *gọi được gì*; dynamic type (JVM, runtime) = *chạy bản nào*.
- **Upcast vs downcast:** `INotify n = new Email()` = upcast (tự động, an toàn); `(Email) n` = downcast (viết tay, rủi ro nổ lúc runtime). Downcast nhiều = abstraction rò rỉ.

## Thực hành
`01 Practice/Polymorphism.java` — 3 thí nghiệm, tự điền TODO và chạy:
1. Vòng lặp `List<INotify>` lẫn Email+Sms, một dòng `n.send(...)` in ra khi EMAIL khi SMS → *thấy* dynamic dispatch.
2. `n.sendBulk()` không cast → compiler chặn (`The method ... is undefined for the type ...`). Chạm ranh giới static type.
3. `((EmailNotification) n).sendBulk(...)` chạy được → hiểu cast đổi static type compiler thấy *cho đúng biểu thức đó*.
- Lỗi gặp phải: gọi `.send()` lên cả `List` thay vì từng phần tử (`... undefined for the type List<INotify>`) → sửa bằng for-each. (Lỗi cú pháp Java, không phải concept.)

## Khái niệm thứ hai — [[Exception Handling]] (checked vs unchecked) — `Understood` (Good)
Dạy tiếp trong cùng buổi theo yêu cầu "học tiếp".
- Bạn tự tách trục: `IOException` = bất định **ngoài tầm kiểm soát** (bên thứ ba / mạng), chắc chắn có ngày xảy ra dù code đúng → **checked**, compiler **ép** `catch`/`throws`; `NullPointerException` = **bug trong logic mình viết**, sửa code là hết → **unchecked**, không ép.
- Nối được chữ "ép" tới stakes ngân hàng: ép xử lý = không cho **nuốt lặng lẽ** một thất bại tất yếu → tránh "trừ bên mình mà chưa cộng bên kia" (trạng thái bất nhất).

## Khái niệm thứ ba — [[Collections Framework - HashMap|HashMap / HashSet & Big-O]] — `Understood` (Good)
Học tiếp theo yêu cầu. Mở bằng tình huống chống trùng 10 triệu ID.
- Tự nêu: `List.contains` = **quét tuyến tính O(n)**; HashMap **tính** vị trí từ key → **~O(1)**, "chỉ một phép tính, không phụ thuộc n".
- Seed cơ chế: key → hàm băm → hash code (số to) → **nén** về index → ô (bucket) trong mảng nội bộ. Bạn hỏi lại đúng chỗ ("ô có phải RAM không", "tên ô có phải hash code không") → làm rõ hash code ≠ index.
- Đụng độ: tự suy ra "cùng ô thì dùng `.equals()` tìm đúng cái" → nắm hai vai trò: **hashCode → nhảy tới ô, equals → chọn đúng cái trong ô**.
- Reward/insight: đó là lý do object làm key phải override **cả** `hashCode()` lẫn `equals()` (hợp đồng) — câu phỏng vấn kinh điển.
- **Chú ý retention:** đây là concept thứ 3 trong buổi, phần đụng-độ + hợp đồng hashCode/equals còn mới → cần soi kỹ ở review 07-08.
