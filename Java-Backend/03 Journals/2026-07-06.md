## Khái niệm hôm nay
### [[Interface vs Abstract Class]] — `Understood` (Good)
- Dạy qua tình huống payment (`CreditCardPayment` / `BankTransfer` / `MoMoPayment`).
- Trục quyết định: **chia sẻ CODE + STATE thật → abstract class** (tốn suất kế thừa duy nhất); **chỉ hứa một khả năng → interface** (gắn được nhiều).
- Cơ chế mấu chốt bạn tự gọi tên: *interface không chứa logic/state nên không có gì đụng nhau → gắn bao nhiêu cũng được; kế thừa kéo theo toàn bộ state+code của cha nên nhiều cha là loạn (diamond problem) → Java chỉ cho 1 `extends`.*
- Hệ quả: "code theo interface" = gốc của [[Dependency Injection]] (Spring/NestJS).

## Thực hành
`01 Practice/Interface_vs_Abstract_Class.java` — bài Notification (Email/Sms).
- Cấu trúc chuẩn: `interface INotify` (hợp đồng) + `abstract class ANotify` (state `recipient` + `formatTimestamp()`) + hai class `extends ANotify implements INotify`.
- **Tự áp DI**: viết class `Context` nhận vào `INotify` (không phải class cụ thể) — code theo interface, không đợi nhắc.
- Sửa cú pháp: dòng `this.recipient = recipient` trong constructor con là **thừa** vì `super(recipient)` đã gán field kế thừa trước đó. Bạn tự chỉ ra và giải thích được `super()` chạy constructor cha trước.

## Feynman Check
> **Hỏi:** Trong một câu, cái gì quyết định chọn interface hay abstract class?
> **Bạn trả lời (lần chốt):** "abstract khi muốn chia sẻ state nhưng mất một lần kế thừa; interface là một bản cam kết có thể sử dụng được method đó mà không phải kế thừa."
> **Đánh giá:** Trúng trục. Tự bỏ được cách nói sai đầu buổi ("nhiều method" / "quy ước chung"). Mức cơ chế → Good.
