### [REVIEW] Collections Framework - HashMap — Good — 2026-07-15
- Hỏi: Nếu chỉ override equals() mà không override hashCode(), cơ chế HashMap lỗi thế nào?
- Trả lời: hashCode() mặc định từ Object (dựa trên địa chỉ bộ nhớ) → hai đối tượng "giống nhau" ra hai hash code khác nhau → nén về hai ô khác nhau → equals() không bao giờ được gọi → trả null.
- Đánh giá: Good — hiểu hợp đồng hashCode/equals: hashCode quyết định ô, equals quyết định entry trong ô.

### [REVIEW] Stream API — Easy — 2026-07-15
- Hỏi: Phần tử số 6 đi qua pipeline filter → map → distinct như thế nào?
- Trả lời: Duyệt dọc: filter → map → distinct → collect trọn gói từng phần tử một. Tiết kiệm RAM (không mảng tạm), CPU cache friendly, short-circuit.
- Đánh giá: Easy — cơ chế lazy evaluation + vertical processing nắm chắc.

### [REVIEW] List - ArrayList vs LinkedList — Easy — 2026-07-15
- Hỏi: Chèn vào đầu danh sách 1000 phần tử — phân tích từng bước.
- Trả lời: ArrayList shift O(n) toàn bộ 1000 phần tử; LinkedList tạo node + sửa 2 con trỏ O(1). LinkedList nhanh hơn.
- Đánh giá: Easy — phân tích chi tiết cơ chế cả hai phía.

## Nội dung mới hôm nay

### Khái niệm mới: ExecutorService và Future
- **Thread Pool**: Giữ sẵn thread cố định, tái sử dụng thay vì tạo/hủy mỗi tác vụ.
- **submit() vs execute()**: submit() trả về Future, exception được gói bên trong; execute() nuốt exception âm thầm.
- **Future**: "Biên lai hẹn giờ" — non-blocking khi submit, blocking khi get(). Gọi get() càng muộn càng tốt để tối ưu song song.
- **Callable vs Runnable**: Callable trả về kết quả và có thể ném exception; Runnable thì không.
- **ExecutionException**: get() bọc exception gốc trong ExecutionException → dùng getCause() để lấy nguyên nhân thật.

### Feynman check
- Hỏi: Gửi 2 service vào pool như thế nào để chỉ mất 1s thay vì 2s?
- Trả lời: submit() cả 2 task vào pool ngay, main thread làm việc khác, rồi get() ở cuối.
- Đánh giá: Easy.
