# Nhật ký buổi học — 2026-07-17 (C)

**Chủ đề:** Java-Backend — Design Pattern: Singleton, Factory, Builder.

## Nội dung mới hôm nay

### Singleton Pattern
- **Vấn đề:** Các đối tượng như `DatabaseConnection` nếu liên tục dùng `new` sẽ mở nhiều kết nối, làm tràn tài nguyên. Cần gom về 1 instance dùng chung.
- **3 thành phần cốt lõi của Singleton:**
  1. `private` constructor: ngăn bên ngoài tự ý `new`.
  2. `private static` biến lưu trữ & `public static` method (thường gọi `getInstance()`) để code bên ngoài lấy instance.
  3. **Thread-safety:** Double-Checked Locking kết hợp `synchronized` và `volatile`.
- **Double-Checked Locking:**
  - Check 1 (không lock): tránh overhead xếp hàng, luồng đi thẳng nếu instance đã tạo.
  - Check 2 (trong lock): tránh race condition cho lượt tạo duy nhất.
- **Vai trò của `volatile`:** Ngăn compiler/CPU "reordering" (đảo lệnh) các bước khởi tạo (cấp vùng nhớ → gán địa chỉ → chạy constructor). Không có `volatile`, luồng khác có thể lấy ra object chưa chạy xong constructor.
- **Bill Pugh Singleton (Static Inner Class):** Lợi dụng đặc tính "chỉ load class khi cần" và "khởi tạo class là thread-safe" của JVM để tạo Singleton an toàn, lười (lazy) mà không cần `synchronized` hay `volatile`.
- **Enum Singleton:** Lợi dụng JVM để đảm bảo đúng 1 instance, an toàn, chống được serialize và reflect. Nhược điểm là Eager.
- **Thực hành:** Hoàn thiện 3 `TODO` (private constructor, volatile, double-checked lock), test đa luồng.

### Factory Pattern
- **Vấn đề:** Khởi tạo object qua nhiều `if-else` lặp lại khắp nơi, khó bảo trì. Caller biết quá sâu về class cụ thể.
- **Giải quyết:** Gom logic tạo object (`switch-case`) vào một class Factory duy nhất. Trả về interface chung. Caller gọi Factory thay vì gọi `new`.
- **Tham số đặc thù:** Các thuộc tính riêng của object con được truyền qua constructor lúc khởi tạo trong Factory. Interface giữ nguyên để đảm bảo Abstraction.
- **Thực hành:** Viết Payment interface, các lớp Visa/Momo/MasterCard, và PaymentFactory.

### Builder Pattern
- **Vấn đề:** Object có nhiều thuộc tính tuỳ chọn → `new` phải truyền `null` mù mờ hoặc phải viết hàng đống Telescoping Constructor.
- **Giải quyết:** Constructor chính private nhận tham số duy nhất là Builder. Builder dùng `set...` trả về `this` (Method chaining) để dồn data. Chốt bằng `.build()`.
- **Thực hành:** Lắp ráp class `Computer` có tham số bắt buộc và tùy chọn.

## Feynman Check

### Bill Pugh Singleton — Easy
- Hỏi: Bill Pugh Singleton không cần `synchronized` hay `volatile`, nhưng vẫn thread-safe và lazy. Cơ chế nào của JVM đảm bảo điều đó?
- Trả lời: Nêu được Lazy Loading + Initialization Lock của JVM class loader, và nguyên tắc Happens-Before tránh đảo lệnh. Đánh giá: Easy.

### Factory & Builder — Good
- Hỏi: Tại sao truyền tham số đặc thù qua constructor trong Factory thay vì method interface? Tại sao method `set...` trong Builder phải `return this`?
- Trả lời: "1 tại vì để bảo đảm interface không bị vỡ và đảm váo nó làm đúng chức năng của nó. 2 this ở đây là để return về object chính và build đòng vai trò return về object đó".
- Đánh giá: Good — nắm được nguyên tắc bảo vệ Abstraction cho Factory. Builder hơi lỡ từ (this trả về Builder, build() trả về object chính) nhưng ý chính hiểu cơ chế tích data.
